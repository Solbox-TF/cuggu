# 2026-02-11 AI 테마 생성 품질 — 6인 팀 분석

## 작업한 내용

AI 청첩장 테마 생성 기능의 품질 문제를 6인 팀(개발자, 디자이너, 기획자, 프롬프트 엔지니어 x3)으로 병렬 분석. 각자 독립적으로 코드베이스 읽고 경쟁사 리서치 후 종합 보고서 작성.

- **개발자**: 렌더링 아키텍처 분석 — BaseTemplate, AnimatedSection, CoverSection, 스키마 전수 조사
- **디자이너**: 빌트인 6개 테마 비평 + 경쟁사 디자인 패턴 분석 + "유령 기능" 발견
- **기획자(PM)**: 경쟁사 기능 갭 분석 + 임팩트×난이도 매트릭스 + 실행 로드맵
- **프롬프트 엔지니어 #1**: 현재 시스템 프롬프트 감사 — 규칙 과잉, anchor bias, 페르소나 무력화 등 10개 문제 도출
- **프롬프트 엔지니어 #2**: 데어무드/바른손/보헴/카드큐/투아워게스트 등 경쟁사 시각적 기능 비교표 + 2024-2025 트렌드
- **프롬프트 엔지니어 #3**: 새 프롬프트 아키텍처 설계 — Mood Archetype 5종, 확장 시드 13축, 파티클/오프닝/전환 스키마 설계

## 왜 했는지

AI 테마 생성 결과물이 "딱딱하고 촌스럽다"는 불만. 경쟁사 대비 시각적 풍부함이 현저히 부족. 이전 세션에서도 시도했지만 context 차서 중단됐고, 결과물에 만족 못해서 완전히 새로 시작.

## 논의/아이디어/고민

**6명 전원 수렴한 핵심 결론**: "프롬프트를 아무리 개선해도 렌더러가 표현할 수 없는 건 나올 수 없다."

- PM: "프롬프트 개선은 10~15% 효과가 한계. 렌더러 확장이 선행되어야."
- 디자이너: "The theme system is a facade — vocabulary of a rich design system, rendering of a minimal one."
- 프롬프트 엔지니어: "금지형 규칙 과잉 → AI가 안전한 기본값으로 수렴. 근데 그건 렌더러 고친 후에 같이 고쳐야 의미 있음."

**파티클 CSS vs Canvas 논쟁**: 개발자는 CSS 추천 (50개까지 성능 OK, 유지보수 단순). Canvas는 100개+ 필요할 때. 현 단계에선 CSS.

**무료 vs 유료**: PM은 "기본 3종 무료 + 프리미엄에서 전종 + 밀도 조절" 모델 추천.

## 결정된 내용

1. **실행 순서**: Quick Win (1~2일) → Phase 0 프롬프트 (1일) → Phase 1 렌더러 (1주) → Phase 2 (1~2주)
2. **Quick Win 5개**: Ken Burns 커버(30분), SVG 디바이더(4h), 폰트 로더(3h), 교차 섹션 배경(1h), 유령기능 제거(2h)
3. **Phase 1 핵심**: 파티클 6종 + 오프닝 3종 + 스크롤 애니메이션 확장 + 섹션 전환
4. **프롬프트 아키텍처**: Mood Archetype 5종 + 시드 13축(레이아웃 7 + 스타일 6) + Mandatory Differentiation + Anti-Patterns

## 느낀 점/난이도/발견

- **6인 병렬 분석이 효과적이었음**. 각 관점에서 독립 분석 → 교차 검증하니 결론이 수렴해서 신뢰도 높음.
- **"유령 기능" 발견이 중요했다**. 스키마에 선언은 되어 있지만 렌더링 코드가 없는 기능들 (borderStyle, animate-fadeIn 등). AI가 생성해도 화면에 안 나옴 → 사용자 신뢰 붕괴.
- **경쟁사 분석이 눈을 뜨게 함**. 데어무드의 오프닝 + 파티클 조합이 시장 기대치를 만들고 있음. "오프닝 없으면 싼티"가 이미 시장 인식.
- **이전 세션의 미커밋 변경 3파일** (theme-prompt.ts, theme-generation.ts, route.ts)은 프롬프트 레벨 개선이라 이번 분석 결론과는 방향이 다름. 렌더러 확장 후에 통합 검토 필요.

## 서랍메모

- Pantone 2025 Mocha Mousse(17-1230) 참고할 것. 경쟁사들도 earthy tone 트렌드.
- 디자이너가 추천한 프리미엄 골드: #C4A265 (현재 classic의 #B8860B는 old web 느낌)
- "Statement Typography" 트렌드: 이름이나 날짜 하나를 oversized로 처리 (text-4xl~6xl + tracking-[0.15em]+)
- 프롬프트에서 "Korean wedding stationery designer"보다 "Kinfolk/Cereal magazine aesthetics, Amore Pacific branding level"이 AI 출력 수준을 올림
- sectionBg에서 greeting/ceremony 키는 실제로 컴포넌트에서 읽지 않음 — 프롬프트에서 이 키 유도하면 안 됨

## 내 질문 평가 및 피드백

- "팀 구성해서 협업해서 분석하고 최선의 방법을 도출해보자" — 좋은 접근. 6인 병렬 분석이 단일 관점보다 확실히 풍부한 결과를 냄.
- "다른 업체의 모바일 청첩장을 보면 화면 전환도 되고 애니메이션도 들어가고" — 핵심 문제를 정확히 짚었음. 결국 6명 전원이 같은 결론으로 수렴.
- "프롬프트 문젠가?" — 프롬프트도 문제지만 근본은 렌더러. 이 질문 덕에 "프롬프트 vs 렌더러" 경계를 명확히 할 수 있었음.
